## 第1章 なぜ今「ループ」が重要なのか

この2年ほど、私たちは「AIにどう指示すればうまく動くか」を工夫してきた。その工夫の重心は、**プロンプト工学（どう指示するか）→ コンテキスト工学（どう文脈を渡すか）→ ハーネス工学（暴走をどう制御するか）→ ループ工学（そもそも指示する仕組みごと設計する）** と移ってきた 🔶（この系譜整理は解説動画による）。ループエンジニアリングという言葉自体は、OpenAIのPeter Steinberger氏らが発信し、Addy Osman氏らが広めたことで一気に話題になった 🔶。ただし公式は、これを「まったく新しい発明」ではなく、**以前からあったエージェント構築の実践に名前がついたもの**として扱っている ✅。

Anthropicは2026年6月30日の記事「Getting started with loops」で、ループを **「エージェントが、決められた停止条件を満たすまで作業のサイクルを繰り返すこと」** と定義した ✅。従来は「入力→出力を読む→また入力」と、人間が一手ずつチェックしていた。これに対しループは、**作業を見つけて割り当て、検証し、完了を記録し、次を決める**という小さなシステムを組み、そこにエージェントを走らせる。人間の役割は「プロンプトを逐一書く人」から **「ループを設計・管理する監督」** へ移る 🔶。

なぜ今かというと、**AIモデル自体が賢くなったこと**と、**暴走を止める“ハーネス”の議論が成熟したこと**の2つが揃ったから 🔶。この2つが土台になって、長時間・自律的にエージェントを走らせても破綻しにくくなった。

ただし公式が最初に釘を刺すのは、**「すべてのタスクに複雑なループは不要。最もシンプルな解決から始め、必要な場面でだけパターンを選べ」** という点 ✅。まずは手でやっている繰り返し作業を1つ、ループにしてみる——それが公式の勧める“はじめの一歩”だ。

- 進化の系譜：Prompt → Context → Harness → **Loop** Engineering 🔶
- ループの定義（公式）：**停止条件を満たすまでサイクルを繰り返す** ✅
- 「思考はアウトソースできても、**理解はアウトソースできない**」——理解せずに回すと“ゴミ生成マシン”になる 🔶

### ループは新発明ではない — 各分野の実践例

「条件を満たすまでエージェントが作業を繰り返す」という試みは、実は各分野で先行して積み重ねられてきた 🔶。それらに名前がついて一般化したのが「ループ」。抽象論に留めず、具体例を並べると腹落ちする（各事例の実体＝論文・リポジトリは一次情報 ✅）。

| 分野 | 実践例 | 中身 |
|---|---|---|
| ソフトウェア開発 | **Ralph Loop**（Geoffrey Huntley, 2025）🔶 | `while :; do cat PROMPT.md \| claude-code; done` でプロンプトを送り続ける。状態は PROMPT.md に持たせ、再起動のたびに続きから。のちにClaude Codeの `/loop`・`/goal` に相当する形へ |
| 自動研究AI | **autoresearch**（Karpathy, 2026）🔶 / **The AI Scientist**（Sakana AI）✅ | 前者は英語で書いた手順書を読ませ `train.py` を反復改善し「勝った変更だけ残す」ループ。後者は着想→実験→執筆→査読までを自動で回す（[arXiv:2408.06292](https://arxiv.org/abs/2408.06292)） |
| 自己検証（学術） | **Self-Refine** / **Reflexion** ✅ ／ **IMO2025 金メダル級** ✅ | 出力に自分でフィードバックして改善する反復（[Self-Refine 2303.17651](https://arxiv.org/abs/2303.17651) / [Reflexion 2303.11366](https://arxiv.org/abs/2303.11366)）。検証→改善パイプラインで数学オリンピック2025を5/6正解（[arXiv:2507.15855](https://arxiv.org/abs/2507.15855), UCLA） |

いずれも「実行 → 検証 → だめならやり直す」という同じ骨格を持つ。Self-Refine・Reflexion・IMOパイプラインは、第3章の**検証（Doer-Verifier）**の先行例とも言える。

## 第2章 4種類のLoopを理解する

「Getting started with loops」は、ループを**4種類**に整理している ✅。それぞれ「何をきっかけに始まり／何で止まり／どの機能を使い／どんなタスク向きか」で性格が違う。

**① ターンベースループ** — きっかけ：人がプロンプトを送る／止まる：Claudeが「完了」or「要追加情報」と判断。短め・非定型のタスク向き。手動でループを回す最も基本の形。`SKILL.md` に検証手順を書いておくと、Claude自身の自己検証が効く ✅。

**② ゴールベースループ（`/goal`）** — きっかけ：その場のプロンプト／止まる：ゴール達成 or 最大試行回数。検証可能な終了基準があるタスク向き。**別の小さなモデルが「完了したか」を毎ターン判定する**ので、書き手が自分で採点しない ✅。例：`ホームページのLighthouseスコアを90以上に。5回試してダメなら停止`。

`/goal` の挙動を分解すると、**実行役（作業する）と判定役（完了を判定する）** の二役になっている。判定役は**別モデルが Hook で起動**して完了条件を判定し（解説動画では既定は小さいモデル＝Haiku と紹介 🔶）、未完了なら実行役にプロンプトを注入し直し、完了ならユーザーに返す。**書き手と採点者を分ける**ことが品質の肝で、これは第3章の Doer-Verifier と同じ発想 ✅。

**③ タイムベースループ（`/loop`・`/schedule`）** — きっかけ：指定した時間間隔／止まる：自分で止める or 作業完了。繰り返し作業・外部システム監視向き ✅。`/loop` は**手元のPCで動く**（PCを閉じれば止まる）。`/schedule` は**クラウドの「ルーティン」**で、マシンを閉じても生き続け、実行のたびにリポジトリを新しくクローンして `claude/` 付きブランチで作業する。cron（最短1時間間隔）／GitHub webhook／専用APIの3通りで起動できる ✅。例：`/loop 5m 自分のPRを見て、レビューコメントに対応し、落ちてるCIを直す`。

**④ プロアクティブループ** — きっかけ：イベント or スケジュール（人がリアルタイムに関与しない）／止まる：個々のタスクはゴール達成で終わり、ルーティン自体は止めるまで継続。バグ報告のトリアージなど、**内容が明確で繰り返す作業**向き。「オートモード」「ダイナミックワークフロー」（研究プレビュー）を組み合わせ、数百のエージェントをオーケストレーションする ✅。

内側と外側の二重構造で捉えると分かりやすい 🔶。**内側のループ**は「1つの作業を、検証しながら完了まで回す」部分。それが何度も回って、**外側のループ**＝「次のトリガーを待って集約し、自律的に回り続ける全体」が形づくられる。ループの成熟度を段階で見る整理（Matt Van Horn 氏）もあり、レベルが上がるほど人間の介入なしに外側のループが自走する 🔶。

公式のまとめ表（何を任せ、いつ使うか）✅：

| ループ | 何を任せるか | 使うタイミング | 主な機能 |
|---|---|---|---|
| ターンベース | 確認作業 | 探索中・検討中 | 検証用スキル |
| ゴールベース | 停止条件 | 「完了」の定義が明確 | `/goal` |
| タイムベース | きっかけ | 外部で定期的に発生 | `/loop`・`/schedule` |
| プロアクティブ | 指示そのもの | 繰り返し発生・内容が明確 | 上記全部＋ダイナミックワークフロー |

- **停止条件が機械で判定できるタスク**ほどループに向く（テストが通る・lintゼロ・スコア達成）✅🔶
- 逆に「センス・感性」で良し悪しが決まるタスク（企画・台本など）は、明確な停止条件を渡しにくく、現状は難しい 🔶

### ループの成熟度：5段階（Matt Van Horn の整理）

「ループを回す」といっても成熟度に幅がある。Matt Van Horn はこれを5段階に整理している 🔶。第1章で挙げた Ralph や、第2章の `/goal` が段階のどこに位置するかが見えると、自分が今どこを目指すのかが掴める。

| Lv | 形 | 中身 |
|---|---|---|
| 1 | ReAct | 思考→ツール呼び出し→結果を読む、を完了まで繰り返す。**人間が監視** |
| 2 | AutoGPT | 目標を与え、自分自身にプロンプトを送らせる（ReActより自律） |
| 3 | Ralph | 毎ターン文脈をリセットし、固定プロンプトを無限に送り続ける |
| 4 | `/goal` | **判定モデル**が「完了」と判定するまで送り続ける |
| 5 | オーケストレーション | **Loopを回すLoop**。人の介在なく作業をトリガーする基盤の確立 |

B. Cherny や P. Steinberger が「ループ（エンジニアリング）」で実際に意味しているのは、**Lv5＝外側のループが自走する状態** 🔶。第2章の「内側／外側のループ」で言えば、外側が自動で回っている段階にあたる。

## 第3章 良いLoopは「検証」がすべて

ループを賢くする鍵は、実行そのものより **検証（verification）** にある。公式の言い方はシンプルで、**「検証が定量的であるほど、Claudeは自己検証しやすい」** ✅。「なんとなく良さそう」ではなく「エラー0件」「テスト全パス」「スコア90以上」のように、機械が判定できる形にするほどループは自走する。

具体的な仕込みどころは次の通り。

1. **SKILL.mdで手動検証を自動化する** — ターンベースで人がやっていた確認（サーバ起動→操作→スクショ→コンソール確認→監査）を `SKILL.md` の手順として書く。「どれか失敗したら直して最初からやり直す。中途半端に返さない」まで書き込むのがコツ ✅。
2. **MCP／Chrome DevTools で“外の現実”を見せる** — Model Context Protocol で外部ツールをつなぎ、Chrome DevTools MCP で Core Web Vitals を計測するなど、**エージェントが自分で結果を観測できる手段**を持たせる ✅。
3. **Doer-Verifier（実行役と検証役を分ける）** — 起案するエージェントと、その成果を採点するエージェントを分けると、自己採点バイアスが減って品質が安定する ✅。`/goal` の「別モデルが完了を判定する」仕組みも、この考え方の実装形。
4. **コードレビューを別Agentに任せる** — 同じ文脈で書いた本人がレビューすると見落としが増える。**新しい文脈の別エージェント**にレビューさせると、偏りのない指摘が得られる ✅。

なぜ検証をここまで重視するのか。ループは基本的に**ブラックボックス化しやすい**からだ 🔶。回している間の細部は人が追えない。だからこそ、前段で完了条件を明確に定義し、要所に検証を挟む。**「思考はアウトソースできても、理解はアウトソースできない」**——検証を怠って回すだけだと、量産されるのは低品質な成果物になりかねない 🔶。

## 第4章 Loopを支えるClaude Code設計

ループの実体は、Claude Code の各機能で組み立てる。Anthropicの記事「Steering Claude Code」は、Claudeの動きを操る方法を**7つ**に整理している（CLAUDE.md／Rules／Skills／Sub Agents／Hooks／Output Style／システムプロンプト追記）✅。大事なのは **「どこに何を書くか」** の使い分けだ。それぞれ「いつ読み込まれるか」「コンテキストをどれだけ食うか」が違う。

| 手法 | 何を書く | 読み込み | ひとことで |
|---|---|---|---|
| **CLAUDE.md** | 事実（ビルドコマンド・構成・規約・チームの約束） | セッション開始時に常駐 | 常に知っておくべき土台。200行以内に絞る ✅ |
| **Rules** | 特定パスの制約（例: API は Zod 必須） | パスに触れた時（scoped）／常時（unscoped） | ファイル固有の約束。`paths` で軽く保つ ✅ |
| **Skills** | 手順・ワークフロー（デプロイ、レビュー、検証） | 呼び出し時に本体をロード | 30行を超える“やり方”はここ。低コスト ✅ |
| **Sub Agents** | 隔離したいサイドタスク（ログ解析・依存監査） | Agentツール呼び出し時 | 別コンテキストで実行し、結果だけ親に返す ✅ |
| **Hooks** | 決定論的なガードレール・自動化 | ライフサイクルのイベントで発火 | 「毎回必ず」「絶対にするな」はここ ✅ |
| **Output Style** | 役割の大きな変更 | システムプロンプトに常駐 | 一般アシスタント化など大改造のときだけ ✅ |

判断の軸はいくつかある ✅。**「毎回必ずフォーマッタをかけたい」「絶対にこのコマンドは打たせない」**——こうした確実性が要るものは、CLAUDE.md や Rules（＝“指示”なので長いセッションやプロンプトインジェクションで破られうる）ではなく、**Hook（`PreToolUse` で exit 2）や権限設定**という決定論的な仕組みで止める。**30行を超える手順**は事実置き場の CLAUDE.md ではなく **Skill** へ。**中間結果を見たくない重い調査**は **Sub Agent** に隔離する（本体コンテキストを汚さず、最終結果だけ戻る）。**Skill と Sub Agent の違い**は「メインスレッドで手順を見せながら進めたいか（Skill）／隔離して結果だけ受け取りたいか（Sub Agent）」で選ぶ ✅。

つまりループ設計とは、**土台＝CLAUDE.md／局所ルール＝Rules／手順＝Skills／隔離作業＝Sub Agents／ガードレール＝Hooks** を適材適所に配置する作業でもある。

## 第5章 Human-Agent Teamへ

最後は、ループを個人の作業から**チームの仕組み**へ広げる話。記事「Building effective human-agent teams」は、人とAIが同じワークスペースで働く「マルチプレイヤー・エージェント」の時代の作法を4つにまとめている ✅。

1. **公開の場で働く（Work in public）** — エージェントは Slack・コード・ドキュメント・議事など**検索できるテキストからしか状況を理解できない**。**「書かれていなければ、存在しない」**。だから意思決定や文脈を、共有の場に残すことが前提になる ✅。
2. **役割を明確に、道具を持たせる（Role設計）** — 「名簿（ロスター）」で誰が何を担うかを明文化し、人もエージェントも同じスレッドで働けるようにする。各エージェントに必要なツール（BigQuery、Playwright MCP など）を渡す ✅。
3. **北極星（North Star）を掲げる** — 事業・ミッションに根ざした野心的な長期ゴールを言語化し、エージェントに共有する。すると優秀なエージェントは、その目標に沿って**自分から次の作業を提案しはじめる**（＝プロアクティブになる）✅。
4. **信頼を時間をかけて積み上げる（Trust building）** — 最初は人がすべての出力をレビューし、フィードバックする。**実績が出たタスク種別から自律範囲を広げる**。Doer-Verifierや週次の「教訓と失敗」レポートで、同じ失敗を繰り返さない仕組みを作る ✅。

公式が強調するのは、これらは**新しい発明ではない**ということ ✅。明確な役割、良いドキュメント、共通の品質基準、失敗からの学習——どれも昔から良いチームがやってきたこと。エージェントが加わることで、それが**“あった方がいい”から“必須”に変わる**。

結論は第1章に戻る。**「AIに書かせる」時代から「AIが改善し続ける仕組みを設計する」時代へ**。人は書く人でも丸投げする人でもなく、**ループを設計し、停止条件を決め、要所で検証する監督**になる。まず1つ、いつも手でやっている繰り返しをループ化してみることから始めよう。

## 第6章 実践：最初のループを回す

ここまでが「理解」。最後は、見終わってすぐ手を動かすための実践パート。手順は「**見極め → 最初の1個 → 失敗/コスト管理 → ロードマップ**」。

### 1. まず「ループ化するタスク」を見極める

いちばん最初の判断がこれ。**完了を機械が判定できるタスク**（テストが通る・lintがゼロ・ビルド成功・スコア90以上）はループに向く。逆に、**センスや感性で良し悪しが決まるタスク**（企画・文章の良し悪しなど）は停止条件を明確に渡せず、現状は難しい 🔶。迷ったら「完了を機械が判定できるか？」で切り分け、**まずシンプルな1つ**から始める ✅。

### 2. 最初の1個をやってみる（例：テスト修正のループ化）

いつも手でやっている「失敗したテストを直す」作業をループにしてみる。

1. **タスクを選ぶ** — 毎回手でやっているテスト修正
2. **停止条件を決める** — 対象テストがすべて green（機械が判定できる）
3. **`/goal` で回す** — `/goal 失敗しているテストが全部通るまで実装を直して。5回試してダメなら理由を添えて停止`
4. **検証で返る** — 別モデルが「完了したか」を判定し、達成でユーザーに返す ✅

この形は配布の「Loop設計テンプレート」にそのまま落とせる。

### 3. つまずきやすい所とコスト管理

実際に回すと最初にぶつかるのがここ ✅（一部は解説動画の指摘 🔶）。

- **停止条件を曖昧にしない** — 無限ループとトークン暴走の元。必ず数値・明確な条件で
- **過剰投入を避ける** — 小さいタスクに複数エージェントやループは不要。安く速いモデルで足りることも多い
- **決まりきった作業はスクリプト化**、ルーティンは必要以上の頻度で回さない
- **`/usage` で使用量を確認**、各ループは隔離環境（worktree / クラウドボックス）で回す
- **出力は必ずレビュー** — 理解はアウトソースできない 🔶

### 4. 導入ロードマップ

| タイミング | やること |
|---|---|
| 今日 | 手作業を1つ選び、`/goal` でループ化してみる |
| 1週間 | 検証手順を `SKILL.md` に、土台を `CLAUDE.md` に整備して自己検証を効かせる |
| 1か月 | `/loop`・`/schedule` で定期化し、実行役と検証役（Doer-Verifier）を分ける |
| その先 | イベント駆動のプロアクティブループ、チーム運用（北極星・名簿）へ |

まず1つ回して、止まる所・行き過ぎる所を観察して直す——小さく始めて反復するのが公式の勧め ✅。

### 5. もう一歩：実践ワークフロー例（計画→並列→検証→PR）

「1個」に慣れたら、複数タスクを束ねる型に広げられる。解説動画で紹介されていた実装フローが分かりやすい 🔶（Matt Pocock のスキルを併用した例）。

1. **計画を詰める** — `Q/A → PRD → ISSUES`。ドキュメントを介してエージェントと認識を合わせ、タスクを独立した issue に切り分ける
2. **issue 単位で並列** — 各 issue を **worktree** に割り当て、衝突しない範囲で並行実装する
3. **テストを先にレビュー** — 各タスクのテストケースを**自然言語で書かせ、実装前に人間がレビュー**（ここが品質の分かれ目）
4. **`/goal` で実装** — テストが通るまでループ（TDD のサイクル）
5. **PR → 人間が承認・マージ**

ポイントは、**人間が「計画・テスト・最終承認」を握り、実装の反復は `/goal` に任せる**という役割分担。内側（実装ループ）は自動、外側（計画と検証）は人間が回す形で、第2章の「内側／外側のループ」の実例になっている 🔶。
