# 🎁 Loop設計テンプレート

ループを「なんとなく回す」のではなく、トリガー・停止条件・検証・記録まで埋めてから回すための穴埋めテンプレート。

**受け取り方**: 下のコードブロック右上のコピーボタンでコピーするか、このファイルをそのままダウンロードしてください。

**使い方**: そのままコピーして、対象タスク・停止条件を埋める。まずは1つの繰り返し作業をループ化することから始める。

```text
# Loop 設計テンプレート

## 0. まず決める（シンプルから始める）
- このタスクは本当にループが必要か？（1回のプロンプトで済むなら不要）
- 使うループの種類: [ ] ターン [ ] ゴール(/goal) [ ] タイム(/loop・/schedule) [ ] プロアクティブ

## 1. 目的（Goal）
- 何を達成したら「完了」か:
- 完了を機械が判定できる形にする:（例: テストが通る / Lighthouse 90以上 / lintゼロ）

## 2. トリガー（いつ始まるか）
- [ ] 人がプロンプトを送る（ターンベース）
- [ ] その場で /goal 実行（ゴールベース）
- [ ] 時間間隔 /loop（ローカル）・/schedule（クラウド常駐）
- [ ] イベント（PR・webhook・メール受信・監視アラート）＝プロアクティブ

## 3. ループ本体（Act → Verify）
1. 実行役: 作業する
2. 検証役: 完了条件を満たしたか判定する（実行役と分ける）
3. 未達なら差分を直して再実行、達成ならユーザーに返す

## 4. 停止条件（Stop condition）※必ず数値・明確な条件で
- [ ] 完了条件を満たした
- [ ] 最大試行回数（例: 5回）に到達
- [ ] コスト上限に到達

## 5. 記録（Memory / Log）
- 実行日時・対象・変更点・判断理由を LOG に追記
- 次回はこの LOG を読んでから始める（同じ失敗を繰り返さない）

## 6. ガードレール
- 「絶対にやってはいけないこと」は指示ではなくフック＋権限で止める
- 隔離環境（worktree / クラウドボックス）で回す
- 人間の最終確認を挟む工程を明記する
```

**ポイント**: 停止条件を必ず数値で決めておくと、無限ループとトークンの暴走を防げる。「思考はアウトソースできても、理解はアウトソースできない」——出力は必ずレビューする。
